## Next Actions (Prioritised)

### Priority 1 — Harden the Dockerfile (do this first, unblocks Phase 5)

The current Dockerfile runs as root and exposes port 80. This must be fixed before container scanning, because Trivy will flag it as a critical issue.

Target state:
- Pinned base image (`nginx:1.25-alpine`)
- `apk update && apk upgrade` for package patching
- Non-root nginx user (UID 101)
- Port changed to 8080
- `HEALTHCHECK` added
- Custom `nginx.conf` to enable security headers

Files to create/update:
- `docker/Dockerfile`
- `docker/nginx.conf`

---

File to update: `.semgrep.yml` (or move to `.semgrep/rules.yml`)

Checkov scans Dockerfiles and Kubernetes manifests for security misconfigurations. Add it to the Semgrep workflow or a separate workflow.

Scan targets:
- `docker/Dockerfile`
- `k8s/` (once manifests exist)

---

### Priority 4 — Phase 5: Container Scanning with Trivy

Add a Trivy step to the GitHub Actions pipeline after the Docker build.

What Trivy checks:
- OS package CVEs in the container
- Application library CVEs
- Dockerfile misconfigurations
- Secret exposure in image layers

Target: Zero CRITICAL or HIGH CVEs before image can be pushed.

---

### Priority 5 — Phase 6: Image Signing with Cosign

Signs the container image after it passes scanning. Proves the image was built by your pipeline and not tampered with.

Steps:
1. Generate Cosign key pair (`cosign generate-key-pair`)
2. Store private key as GitHub Secret
3. Add Cosign sign step after Trivy passes
4. Add Cosign verify step in deployment job

This is supply chain security — one of the most valued skills in DevSecOps roles.

---

### Priority 6 — Phase 7-8: Kubernetes Manifests + Secure Deployment

Create K8s manifests in `k8s/base/`:
- `deployment.yaml` — app deployment with security context
- `service.yaml` — ClusterIP service
- `networkpolicy.yaml` — deny-all ingress/egress except what's needed

Security context requirements per Pod Security Standards (restricted profile):
```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 101
  readOnlyRootFilesystem: true
  allowPrivilegeEscalation: false
  capabilities:
    drop: [ALL]
```

### Priority 7 — Phase 10: OPA Gatekeeper Policies

Policy as Code — enforce rules at the Kubernetes admission layer.
Rules to enforce:
- All images must be signed (Cosign)

- Images must come from approved registries only

## DevSecOps Concepts Reference

### Shift-Left Security
Moving security checks as early as possible in the development lifecycle — into the developer's local workflow, not just production. Your Gitleaks pre-commit hook is a perfect example.

### SAST (Static Application Security Testing)
Analysing source code without executing it. Finds: injection flaws, hardcoded secrets, insecure patterns. Your tool: Semgrep.

### SCA (Software Composition Analysis)
Scanning third-party libraries for known CVEs. For Node.js: `npm audit`. For containers: Trivy scanning the OS layer.

### Container Hardening
Reducing the attack surface of a container:
- Minimal base image (Alpine over Ubuntu/Debian)
- No unnecessary packages
- Non-root user
- Read-only filesystem where possible
- Drop all Linux capabilities

### Supply Chain Security
Proving that what you built is what gets deployed. Cosign image signing ensures: if someone tampers with your image after it's built, the signature verification fails and deployment is blocked.

### Policy as Code
Expressing security requirements as machine-readable rules (OPA/Rego) instead of documentation. The policy engine enforces them automatically — humans can't bypass them by accident.

### SARIF (Static Analysis Results Interchange Format)
A standard JSON format for security tool output. Uploading SARIF to GitHub lets you see all security findings in one place (Security tab → Code Scanning) regardless of which tool found them.

### Defence in Depth (applied to CI/CD)
```
Layer 1: Local  → Gitleaks (pre-commit)
Layer 2: CI/CD  → TruffleHog + Semgrep + Trivy
Layer 3: Registry → Cosign signature verification
Layer 4: K8s    → OPA Gatekeeper admission control
Layer 5: Runtime → Falco runtime threat detection
```
