## What Was Done Today

### B1c — GitHub Actions Permissions Locked Down

**What:** Added explicit `permissions:` blocks to all four workflow files.

**Why:** By default GitHub gives every workflow write access to the repo. If a compromised GitHub Action or malicious dependency runs inside a workflow, it can push code, delete branches, or read secrets. Locking permissions to the minimum removes that attack surface.

**What each workflow got:**

| Workflow | Permissions | Reason |
|---|---|---|
| `security.yml` | `contents: read` | Only reads code to run smoke test |
| `trufflehog.yml` | `contents: read` + `security-events: write` | Reads full history, writes findings to Security tab |
| `semgrep.yml` | Already had correct permissions | `actions: read` needed for SARIF association |
| `pipeline.yml` | `contents: read` + `packages: write` | Only reason for write is pushing to GHCR |

**Real world relevance:** In 2023 the `tj-actions/changed-files` GitHub Action was compromised. Repos with default write permissions had secrets printed in workflow logs. Repos with locked permissions were safe.

### B2 — Custom Semgrep Rules + Branch Protection

#### Custom Semgrep Rules

**What:** Created `.semgrep.yml` in the project root with 5 custom rules targeting the quiz-site JavaScript codebase.

| Rule | What it detects | Severity |
|---|---|---|
| `no-eval` | `eval()` usage — arbitrary code execution risk | ERROR |
| `no-innerhtml` | `innerHTML` assignments — XSS risk | WARNING |
| `hardcoded-credential` | Variables named password/token/api_key assigned a string | ERROR |
| `weak-random` | `Math.random()` — not cryptographically secure | WARNING |
| `no-console-log` | `console.log()` left in production code | INFO |

**Results:** 9 findings detected in quiz-site/ — all innerHTML and Math.random patterns. These are now visible in the GitHub Security tab permanently and will be rescanned on every push.

**Important lesson on XSS:** Blocking `innerHTML` alone does not prevent XSS. Full XSS prevention requires: input validation + output encoding + Content Security Policy headers. The Semgrep rule catches a common pattern quickly but is one layer of many.

**Updated semgrep.yml workflow** to add `--config=.semgrep.yml` alongside `--config=auto` so custom rules run in CI alongside the 222 community rules.

#### Branch Protection

**What:** Enabled branch protection on `main` with three rules:
- Require pull request before merging
- Require status checks to pass (TruffleHog `scan` job + Semgrep `semgrep` job)
- Do not allow bypassing — applies to admins too

**What each rule does:**
- **Require PR:** No direct push to main. Every change is visible, reviewed, and triggers scans.
- **Require status checks:** TruffleHog and Semgrep must pass before merge button is active. Failed scan = blocked merge.
- **No bypass:** Even admin accounts cannot override the rules. Insider threats and compromised admin accounts cannot sneak around the gates.

**First real proof it worked:** Tried to push directly to main after enabling — got `GH006: Protected branch update failed`. That error is the security working correctly.

**Required approvals set to 0** for solo project — PR is still required but no reviewer needed.

**New workflow from this point:** Create feature branch → push → open PR → wait for scans → merge.

### B3 — Checkov IaC Scanning

**What:** Created `.github/workflows/checkov.yml` — scans `docker/Dockerfile` and `k8s/` YAML against CIS benchmark security rules.

**Why Checkov:** Unlike Semgrep which scans JavaScript, Checkov understands the meaning of Dockerfile and Kubernetes YAML. It knows that a Deployment with no `runAsNonRoot` is a security problem. One tool for all IaC types.

**Findings detected (before B4 fix):**

On `docker/Dockerfile`:
- `CKV_DOCKER_2` — No HEALTHCHECK defined
- `CKV_DOCKER_3` — Container runs as root

Both findings now visible in GitHub Security tab.

**`soft_fail: true`** is set — Checkov reports violations but does not fail the pipeline yet. This becomes a hard gate in B12 at the end of Phase B after all violations are fixed.

**`if: always()`** on the upload step — ensures SARIF is uploaded even if Checkov found violations. Without this, the findings would never reach the Security tab.

### B4 — Dockerfile Hardened

**What:** Replaced the 3-line Phase A Dockerfile with a fully hardened version. Created `docker/nginx.conf` with security headers.

**Before (Phase A):**
```dockerfile
FROM nginx:alpine
COPY quiz-site /usr/share/nginx/html
EXPOSE 80
```

**After (Phase B):**
- Pinned base image: `nginx:1.25-alpine`
- `apk update && apk upgrade` — patches all OS packages at build time
- Non-root user created: nginx (UID 101, no login, no home directory)
- File ownership set so nginx user can access html, cache, logs, pid
- Custom `docker/nginx.conf` copied in with security headers
- `USER nginx` — all subsequent operations and the running process use non-root
- Port changed from 80 to 8080 (unprivileged, non-root can bind)
- HEALTHCHECK added — Kubernetes can now detect and recover from failures

**What each change fixes:**

| Change | Vulnerability it closes |
|---|---|
| Pinned version | Unpredictable upstream changes, reproducibility |
| apk upgrade | Known CVEs in unpatched OS packages |
| Non-root user | Container escape = root access |
| Port 8080 | Root required to bind port 80 |
| HEALTHCHECK | K8s cannot detect frozen or crashed app |
| server_tokens off | nginx version exposed to attackers in headers |
| Security headers | Clickjacking, MIME sniffing, XSS, referrer leakage |

**Verified working:**
```
HTTP/1.1 200 OK
Server: nginx                          ← no version number
X-Frame-Options: SAMEORIGIN
X-Content-Type-Options: nosniff
X-XSS-Protection: 1; mode=block
Referrer-Policy: strict-origin-when-cross-origin
Content-Security-Policy: default-src 'self'; ...
```

```bash
docker exec quiz-hardened whoami
# nginx   ← confirmed non-root
```

**Bug encountered and fixed:** Do NOT put `user nginx;` at the top of `nginx.conf`. The Dockerfile `USER nginx` already switches to non-root. Having both causes nginx workers to silently fail — they start but do not accept connections. Symptom: `curl: (56) Recv failure: Connection reset by peer`. Fix: remove `user nginx;` from nginx.conf entirely.

**Second bug encountered:** Dockerfile edits were not saved in VS Code before running `docker build`. Docker used the cached old 3-line Dockerfile. Symptom: `docker exec quiz-hardened cat /etc/nginx/nginx.conf` showed the default nginx config. Fix: always save files before building. Use `docker build --no-cache` when config files change.

## Key Concepts Learned Today

**Why specific job names matter in branch protection:** GitHub searches through actual job names from workflow run history. `scan` comes from `jobs: scan:` in trufflehog.yml. Without specifying names, the protection rule is decorative — it looks enforced but blocks nothing.

**`soft_fail: true` vs hard gate:** Checkov with soft_fail reports everything but never blocks. Removing it in B12 makes it a real gate. The pattern: see all violations first, fix them, then enforce.

**Docker layer caching:** Docker caches every build step. If a file changes but Docker does not know about it (because nothing in the layer above changed), it uses the old cached layer. `--no-cache` forces a full rebuild. Important for config file changes.

**Content Security Policy is the real XSS defence:** The `Content-Security-Policy` header tells the browser to only execute scripts from your own domain. Even if an attacker injects a `<script>` tag, the browser refuses to run it. This works at the browser level, independent of your application code.

## Tomorrow — B5: Trivy Container Scanning

Trivy scans the built image for CVEs before it is pushed to GHCR. The pipeline will fail if CRITICAL or HIGH CVEs are found — the image never reaches the registry.

Position in pipeline:
```
Build image → Trivy scan → (pass) → Push to GHCR
                         → (fail) → Pipeline stops, image discarded
```

The hardened Dockerfile from B4 should result in significantly fewer CVEs than the Phase A image. Seeing that difference is the concrete proof that container hardening works.
