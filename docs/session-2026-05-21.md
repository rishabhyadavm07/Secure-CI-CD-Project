## What Was Done Today

### B5 ŌĆö Trivy Container Scanning

**What:** Added Trivy scanning to `pipeline.yml` between the build and push steps.

**Why:** Without scanning, vulnerable images reach GHCR and get deployed. Trivy checks every OS package and library in the built image against the National Vulnerability Database. If CRITICAL or HIGH CVEs are found, the pipeline fails and the image is never pushed.

**Pipeline position:**
```
Build image ŌåÆ Trivy scan ŌåÆ (pass) ŌåÆ Push to GHCR
                         ŌåÆ (fail) ŌåÆ Pipeline stops, image discarded
```

**Key settings:**
- `severity: CRITICAL,HIGH` ŌĆö gates on the two most dangerous levels only. MEDIUM/LOW reported but do not block.
- `exit-code: 1` ŌĆö the hard gate. CVEs found = pipeline fails.
- `if: always() && github.event_name == 'push'` on SARIF upload ŌĆö uploads findings to GitHub Security tab only on push to main, not on PRs (GitHub blocks SARIF uploads from PR workflows for security reasons).

**Result:** Zero CRITICAL or HIGH CVEs found on the hardened nginx:1.25-alpine image. The `apk upgrade` in B4 already patched the known vulnerabilities.

**Issues encountered:**
- Initial setup had build and push in a single step (`push: true`). Had to split into separate Build step (`push: false`) and a separate Push step so Trivy could run between them.
- SARIF upload failed on PRs with "Resource not accessible by integration" ŌĆö fixed by adding `github.event_name == 'push'` condition.

### B6 ŌĆö Cosign Image Signing

**What:** Added cryptographic signing to every image pushed to GHCR. Added verification before deployment. The cluster can now reject any image that was not built by your pipeline.

**Why (supply chain security):** After Trivy passes and the image is in GHCR, nothing previously prevented an attacker with GHCR write access from pushing a backdoored image with the `latest` tag. Cosign attaches a cryptographic signature to the image. Verification fails if even one byte was changed after signing.

**Real world relevance:** The SolarWinds attack (2020) inserted malicious code after code review by compromising the build process itself. 18,000 organisations deployed the backdoored software because there was no mechanism to verify build integrity. Cosign solves this at the image level.

**Setup:**
- Generated key pair with `cosign generate-key-pair` in WSL2
- `cosign.key` added to `.gitignore` ŌĆö never committed
- `cosign.pub` committed to the repo ŌĆö anyone can verify
- `COSIGN_PRIVATE_KEY` and `COSIGN_PUBLIC_KEY` added as GitHub Secrets

**Pipeline addition:**
```
Push to GHCR ŌåÆ Install Cosign ŌåÆ Sign image ŌåÆ Verify signature ŌåÆ Print reference
```

**Issues encountered and fixed:**

1. `cosign: command not found` ŌĆö GitHub Actions runner does not have Cosign installed. Fixed by adding `sigstore/cosign-installer@v3` step before signing.

2. `user declined the prompt` ŌĆö Cosign tried to upload to Sigstore's public transparency log and asked for interactive confirmation. CI cannot answer interactive prompts. Fixed by adding:
   - `--tlog-upload=false` to sign command
   - `--insecure-ignore-tlog` to verify command
   Key-based signing does not require the transparency log.

3. `COSIGN_PASSWORD` cannot be empty in GitHub Secrets ŌĆö removed the password env var entirely since no passphrase was set on the key.

4. Repeated `fetch first` push rejections ŌĆö caused by clicking "Update branch" on GitHub PRs which adds a remote commit not present locally. Fixed each time with `git pull --rebase origin branch-name`. Going forward: use `git fetch origin main && git rebase origin/main` locally instead of clicking Update branch on GitHub.

**Key concept ŌĆö `--tlog-upload=false` vs keyless signing:**
The Sigstore transparency log is designed for keyless signing where there is no private key to prove identity ŌĆö GitHub Actions' OIDC identity is used instead and logged publicly. With key-based signing (which we use), the private key IS the proof of identity. The transparency log adds nothing and requires interactive confirmation. Enterprise teams using key-based signing always skip it.

### B7 ŌĆö Dependabot

**What:** Created `.github/dependabot.yml` configuring Dependabot to watch the Dockerfile and all GitHub Actions workflows weekly.

**Why:** Without automated dependency scanning, CVEs in base images and GitHub Actions accumulate silently. The gap between a CVE being published and your system being patched (mean time to patch) grows to weeks or months. Enterprise SLAs require CRITICAL CVEs patched within 24 hours ŌĆö impossible without automation.

**What Dependabot watches:**
- `docker` ecosystem in `/docker` ŌĆö watches `nginx:1.31-alpine` for newer patched versions
- `github-actions` ecosystem in `/` ŌĆö watches all Actions used in workflows

**Immediate result:** Within 2 minutes of merging the Dependabot config, 4 PRs were automatically opened:

|---|---|
| bump nginx from 1.25-alpine to 1.31-alpine | Base image 6 minor versions behind |
| bump actions/checkout from 4 to 6 | 2 major versions behind |
| bump docker/build-push-action from 5 to 7 | 2 major versions behind |
| bump docker/login-action from 3 to 4 | 1 version behind |

All 4 had green security checks. All 4 merged. nginx updated to 1.31-alpine ŌĆö Trivy rescanned and passed.

## Key Learnings Today

**Why split build and push:** The `docker/build-push-action` with `push: true` builds and pushes in one step. Trivy needs to scan the built image before it is pushed. Solution: build with `push: false`, run Trivy, then push separately with `docker push`.

**GitHub Actions SARIF restriction on PRs:** GitHub deliberately blocks SARIF uploads from PR workflows to prevent malicious PRs from poisoning the security results. Solution: add `github.event_name == 'push'` condition so uploads only happen after merge to main.

**Transparency log vs key-based signing:** Sigstore's tlog is for keyless signing (identity proven via OIDC, no stored key). Key-based signing (our approach) uses the private key as proof ŌĆö tlog adds nothing and causes interactive prompts in CI. Always use `--tlog-upload=false` with key-based signing.

**Dependabot works immediately:** PRs appear within minutes of enabling it. All 4 PRs had green checks because the full pipeline ran on them. This is the value ŌĆö security updates validated and ready to merge with zero manual investigation.
