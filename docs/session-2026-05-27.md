## What Was Done Today

### B8 — Kubernetes Security Contexts

**What:** Hardened `k8s/base/deployment.yaml` with pod-level and container-level security contexts.

**Why:** By default, Kubernetes runs containers with whatever permissions the image was built with. A container running as root with full Linux capabilities, a writable filesystem, and no resource limits is a significant attack surface. If an attacker achieves remote code execution inside the container, they can write malware to disk, escalate privileges, and potentially break out to the node.

**Changes made:**

Pod-level `securityContext`:
```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 101        # nginx user from Dockerfile
  runAsGroup: 101
  fsGroup: 101
  seccompProfile:
    type: RuntimeDefault
```

Container-level `securityContext`:
```yaml
securityContext:
  allowPrivilegeEscalation: false
  readOnlyRootFilesystem: true
  capabilities:
    drop:
      - ALL
```

Resource limits:
```yaml
resources:
  requests:
    cpu: "50m"
    memory: "64Mi"
  limits:
    cpu: "100m"
    memory: "128Mi"
```

`emptyDir` volumes for nginx writable dirs (required by `readOnlyRootFilesystem: true`):
- `/var/cache/nginx` — nginx temp files
- `/var/run` — nginx.pid
- `/tmp` — general scratch

**Bug fixed:** Original `deployment.yaml` had `containerPort: 80` but the Dockerfile runs nginx on port `8080` (non-root port). Also fixed `targetPort` in `service.yaml` from `80` to `8080`.

**Key concept — why `readOnlyRootFilesystem` matters:**
An attacker with RCE in a container with a writable filesystem can download tools, write scripts, and establish persistence. With `readOnlyRootFilesystem: true`, they cannot write anything to disk — the attack is contained to memory only.

### B9 — Kubernetes Network Policies

**What:** Created `k8s/base/network-policy.yaml` with two policies:
1. `default-deny-all` — denies all ingress and egress across the entire `quiz-app` namespace
2. `allow-ingress-8080` — allows inbound TCP traffic on port 8080 to quiz-site pods only

**Why:** Kubernetes has a flat network by default — every pod can reach every other pod across all namespaces. If any pod in the cluster is compromised, it can freely communicate with your application. Network Policies create a firewall at the pod level.

**Why no egress allow:**
The quiz-site serves static HTML/JS files from disk. It makes zero outbound connections. Denying all egress is correct — nginx has no reason to call external services.

**Important note on NodePort traffic:**
Traffic arriving via NodePort comes from the node's network stack (via kube-proxy), not from a specific pod. This is why `allow-ingress-8080` uses an empty `from` (allow from any source) rather than a pod selector. The network policy still has value because it blocks all other ports — only 8080 is reachable.

**Result:** App remained accessible at `localhost:30080` after applying both policies.

### B10 — RBAC (Role-Based Access Control)

**What:** Created `k8s/base/rbac.yaml` with three resources:
1. `ServiceAccount: quiz-site` with `automountServiceAccountToken: false`
2. `Role: quiz-site-role` with empty rules (no permissions)
3. `RoleBinding: quiz-site-rolebinding` connecting the account to the role

Updated `deployment.yaml` to reference `serviceAccountName: quiz-site`.

**Why:** Every pod runs under the `default` ServiceAccount by default. This account can have broader cluster permissions than needed, and Kubernetes automatically mounts an API token inside every container. Your nginx process has no reason to talk to the Kubernetes API — so it should not have a token at all.

**`automountServiceAccountToken: false` — what it does:**
Without this, Kubernetes mounts a file at `/var/run/secrets/kubernetes.io/serviceaccount/token` inside every container. An attacker with access to the container can use this token to query the Kubernetes API. Setting this to false removes the token entirely.

**Empty Role — why create it at all:**
A ServiceAccount with no RoleBinding has the default cluster permissions (which vary by cluster). An explicit empty Role and RoleBinding makes the intent clear and auditable — this service account has zero API permissions by design, not by accident.

### B11 — Checkov Kubernetes SARIF Upload

**What:** Updated `.github/workflows/checkov.yml` to upload the Kubernetes manifest scan results (`checkov-k8s.sarif`) to the GitHub Security tab. Also added `github.event_name == 'push'` guard to both upload steps.

**Why:** The k8s scanning step already existed but the results were silently discarded — only the Dockerfile SARIF was uploaded. Kubernetes misconfigurations (missing security contexts, missing resource limits, privileged containers) were being caught by Checkov but never surfaced to the developer.

**The `github.event_name == 'push'` guard:**
GitHub blocks SARIF uploads from PR workflows (to prevent malicious PRs from polluting the Security tab). Without this condition, the upload step fails on every PR. Adding the condition means findings only appear in the Security tab after merging to main — which is the correct behaviour.

### B12 — Pod Security Admission

**What:** Labelled the `quiz-app` namespace to enforce the `restricted` Pod Security Standard:
```yaml
labels:
  pod-security.kubernetes.io/enforce: restricted
  pod-security.kubernetes.io/enforce-version: v1.31
```

**Why:** Security contexts in `deployment.yaml` are a declaration — they can be accidentally removed or overridden. PSA enforces security standards at the **cluster level**. Kubernetes itself rejects any pod that does not meet the restricted bar before it starts, regardless of what is in the YAML.

**What restricted enforces:**
- `runAsNonRoot: true` required
- `allowPrivilegeEscalation: false` required
- `capabilities.drop: [ALL]` required
- `seccompProfile: RuntimeDefault` or `Localhost` required
- `hostPath` volumes blocked entirely
- Dangerous volume types blocked

**What happened when applying:**
Kubernetes immediately warned that the existing pod (without `seccompProfile`) violated the new policy:
```
Warning: existing pods in namespace "quiz-app" violate the new PodSecurity enforce level "restricted:v1.31"
Warning: quiz-site-6c4f9f549c-w5hs2: seccompProfile
```
After applying the updated `deployment.yaml` (with `seccompProfile: RuntimeDefault` added), the new pod came up cleanly with no warnings.

**`enforce-version: v1.31` — why pin the version:**
Without pinning, the restricted standard silently changes when you upgrade Kubernetes. Pinning ensures your policy stays stable across upgrades — you opt into new checks deliberately.

### B13 — ResourceQuota and LimitRange

**What:** Created `k8s/base/resource-quota.yaml` with:

`ResourceQuota` — namespace-level hard caps:
```
pods: 5          services: 3
requests.cpu: 250m    limits.cpu: 500m
requests.memory: 320Mi   limits.memory: 640Mi
```

`LimitRange` — per-container defaults and maximums:
```
defaultRequest: 50m CPU / 64Mi memory
default (limit): 100m CPU / 128Mi memory
max: 200m CPU / 256Mi memory
```

**Why ResourceQuota:**
Individual pod resource limits (set in B8) only protect against one pod misbehaving. ResourceQuota caps the total consumption of the entire namespace. Without it, scaling the deployment to 10 replicas would consume 10× the resources — potentially starving other workloads on the cluster.

**Why LimitRange:**
If a developer deploys a new pod without setting `resources`, it runs unconstrained and is not counted against the ResourceQuota (because it has no requests/limits). LimitRange automatically injects defaults, closing this gap.

**Result from `kubectl describe resourcequota`:**
```
Resource          Used    Hard
limits.cpu        100m    500m
limits.memory     128Mi   640Mi
pods              1       5
requests.cpu      50m     250m
requests.memory   64Mi    320Mi
services          1       3
```
The running pod is already tracked. Real-time enforcement confirmed.
