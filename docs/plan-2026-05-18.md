# Session Plan — 2026-05-18

**Project:** Secure CI/CD Pipeline (Windows 11 / WSL2 / Kali Linux)
**Goal:** Build a fully working CI/CD pipeline first — then systematically secure it, learning *why* each security layer exists by seeing what breaks without it.

---

## Strategic Approach Decision (Made Today)

### Build First. Secure Second.

The project will follow a two-phase strategy:

**Phase A — Build (current focus)**
Get a real, working CI/CD pipeline running end-to-end. No security hardening yet. The pipeline should: build the Docker image, push it to a registry, and deploy it to Kubernetes. It will work, but it will be intentionally insecure.

**Phase B — Secure (next focus)**
Go back through every layer and harden it. Each time you add a security control, you will be able to see exactly what it protects against — because you already built the thing it's protecting.

### Why This Is the Right Approach

- You learn what the pipeline *does* before you learn why it needs to be secured
- Every security fix has a visible before/after — you can point at the exact vulnerability you closed
- This mirrors real-world work: most pipelines you inherit at a job are already built but insecured — your job is to harden them
- Much stronger portfolio story: "I built a pipeline, found its weaknesses, and secured it layer by layer" vs. "I followed a secure setup guide"

## Phase A — Build Roadmap (complete the working pipeline)

### Step A1 — Make the Dockerfile build cleanly

The current Dockerfile is fine for Phase A. It works. It's just not hardened yet — that's intentional.

Verify it builds:
```bash
docker build -t quiz-site:local -f docker/Dockerfile .
docker run -d -p 80:80 quiz-site:local
# Open http://localhost:80 — confirm quiz loads
```

---

### Step A2 — Add Docker Build + Push to GitHub Actions

Extend the main pipeline to actually build and push the image on every push to main.

Registry choice (pick one):
- **GitHub Container Registry (GHCR)** — free, no extra account, integrates with GitHub Actions natively. Recommended.
- **Docker Hub** — works fine, requires Docker Hub account + secret

Add to `.github/workflows/security.yml` (or a new `pipeline.yml`):
```yaml
- name: Build Docker image
  run: docker build -t ghcr.io/${{ github.repository }}/quiz-site:${{ github.sha }} -f docker/Dockerfile .

- name: Push to GHCR
  run: docker push ghcr.io/${{ github.repository }}/quiz-site:${{ github.sha }}
```

GitHub Actions already has permission to push to GHCR with `GITHUB_TOKEN` — no extra secrets needed.

---

### Step A3 — Create Kubernetes Manifests (unsecured, functional)

Create the minimum manifests needed to deploy the app to local Kubernetes.

Files to create:
- `k8s/base/deployment.yaml` — runs the container, no security context yet
- `k8s/base/service.yaml` — exposes the app on a NodePort so you can reach it in the browser
- `k8s/base/namespace.yaml` — dedicated namespace for the app

These will be intentionally basic — no `securityContext`, no resource limits, no network policies. That's the point. You'll see exactly what's missing when you run security scanners against them in Phase B.

---

### Step A4 — Add Kubernetes Deploy Step to GitHub Actions

After the image is pushed to GHCR, the pipeline should deploy it to the cluster.

```yaml
- name: Deploy to Kubernetes
  run: |
    kubectl set image deployment/quiz-site quiz-site=ghcr.io/${{ github.repository }}/quiz-site:${{ github.sha }}
    kubectl rollout status deployment/quiz-site
```

For local Docker Desktop Kubernetes, this requires the runner to have `kubectl` configured — this step may need to be run manually first, then automated.

## Phase B — Secure Roadmap (harden everything, layer by layer)

Each step below fixes a real vulnerability introduced in Phase A. The learning comes from doing them in order — each one has a "before" state you can demonstrate.

### Security Layer 1 — Harden the Dockerfile

**What's wrong in Phase A:** Container runs as root, uses unpatched packages, exposes port 80.
**What you'll fix:**
- Pin the base image to a specific digest
- Run `apk update && apk upgrade`
- Create and switch to a non-root nginx user (UID 101)
- Change EXPOSE to 8080
- Add HEALTHCHECK
- Add a custom `nginx.conf` with security headers (X-Frame-Options, CSP, etc.)

**What Trivy will show before vs. after:** CVE count drops significantly. Root-user finding disappears.

---

### Security Layer 2 — Container Scanning with Trivy

**What's wrong in Phase A:** Images are pushed with no vulnerability check.
**What you'll add:** Trivy scans the image after build. Pipeline fails if CRITICAL CVEs found.

This is where your basic Dockerfile gets exposed. Trivy will show you every CVE in the unpatched base image. You fix the Dockerfile, re-scan, and see the count drop. That's the learning moment.

---

### Security Layer 3 — IaC Scanning with Checkov

**What's wrong in Phase A:** Kubernetes manifests have no security context, no resource limits, no network policies.
**What you'll add:** Checkov scans the `k8s/` manifests and fails the pipeline if CIS benchmark violations are found.

---

### Security Layer 4 — Expand Custom Semgrep Rules

**What's wrong in Phase A:** Semgrep runs with auto rules only.
**What you'll add:** Custom rules targeting your specific codebase:
- `eval()` usage
- `innerHTML` XSS patterns
- Hardcoded credentials
- Weak randomness (`Math.random()`)
- Debug statements left in production code

---

### Security Layer 5 — Image Signing with Cosign

**What's wrong in Phase A:** Anyone could push a fake image to GHCR and it would deploy.
**What you'll add:** Cosign signs the image after Trivy passes. Deployment step verifies the signature before applying. Unsigned images are rejected.

This is supply chain security — one of the highest-value skills in DevSecOps.

---

### Security Layer 6 — Kubernetes Security Contexts

**What's wrong in Phase A:** Pods run as root, can write to filesystem, have full Linux capabilities.
**What you'll add:**
```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 101
  readOnlyRootFilesystem: true
  allowPrivilegeEscalation: false
  capabilities:
    drop: [ALL]
```

---

### Security Layer 7 — Network Policies

**What's wrong in Phase A:** Any pod can talk to any other pod in the cluster.
**What you'll add:** `NetworkPolicy` manifests that enforce deny-all by default, then allow only what's needed.

---

### Security Layer 8 — OPA Gatekeeper (Policy as Code)

**What's wrong in Phase A:** Security rules only exist as YAML comments and documentation — nothing enforces them.
**What you'll add:** OPA Gatekeeper admission controller. Policies written in Rego that the cluster enforces at deploy time. A deployment violating the rules is rejected before it ever runs.

---

### Security Layer 9 — Monitoring (Prometheus + Grafana)

**What's wrong in Phase A:** You have no visibility into what's happening in the cluster.
**What you'll add:** Prometheus scrapes metrics, Grafana visualises them. A security dashboard showing: failed deployments, policy violations, CVE counts over time.

## DevSecOps Concepts Reference

### Shift-Left Security
Moving security checks earlier in the pipeline — into the developer's local workflow. The later a bug is found, the more expensive it is to fix. Your Gitleaks pre-commit hook is shift-left. Running Trivy only in production is not.

### SAST (Static Application Security Testing)
Scanning source code without running it. Finds: injection flaws, hardcoded secrets, insecure patterns. Your tool: Semgrep.

### SCA (Software Composition Analysis)
Scanning third-party libraries for known CVEs. For containers: Trivy scanning the OS and app layers.

### Container Hardening
Reducing the attack surface of a container image:
- Minimal base image
- No unnecessary packages
- Non-root user
- Read-only root filesystem
- Drop all Linux capabilities, add back only what's needed

### Supply Chain Security
Proving that what you built is what gets deployed. If an attacker tampers with your image in the registry, Cosign signature verification will fail and the deployment will be blocked.

### Policy as Code
Security rules expressed as code (OPA/Rego) and enforced automatically by the platform. Not a wiki page. Not a checklist. A machine that says "no" when the rule is violated.

### SARIF (Static Analysis Results Interchange Format)
Standard JSON format for security tool output. Uploading SARIF to GitHub centralises all findings in the Security tab regardless of which tool produced them.

### Defence in Depth (full target architecture)
```
Layer 1: Local        → Gitleaks pre-commit hook
Layer 2: CI/CD        → TruffleHog + Semgrep + Trivy + Checkov
Layer 3: Registry     → Cosign image signing + verification
Layer 4: K8s Admission → OPA Gatekeeper policies
Layer 5: K8s Runtime  → Security contexts + network policies
Layer 6: Observability → Prometheus + Grafana + Falco
```

## DevSecOps Concepts Reference

### Shift-Left Security
Moving security checks as early as possible in the development lifecycle — into the developer's local workflow, not just production. Your Gitleaks pre-commit hook is a perfect example.

### SAST (Static Application Security Testing)
Analysing source code without executing it. Finds: injection flaws, hardcoded secrets, insecure patterns. Your tool: Semgrep.

### SCA (Software Composition Analysis)
Scanning third-party libraries for known CVEs. For Node.js: `npm audit`. For containers: Trivy scanning the OS layer.

### Container Hardening
Reducing the attack surface of a container:
- Minimal base image (Alpine over Ubuntu/Debian)
- No unnecessary packages
- Non-root user
- Read-only filesystem where possible
- Drop all Linux capabilities

### Supply Chain Security
Proving that what you built is what gets deployed. Cosign image signing ensures: if someone tampers with your image after it's built, the signature verification fails and deployment is blocked.

### Policy as Code
Expressing security requirements as machine-readable rules (OPA/Rego) instead of documentation. The policy engine enforces them automatically — humans can't bypass them by accident.

### SARIF (Static Analysis Results Interchange Format)
A standard JSON format for security tool output. Uploading SARIF to GitHub lets you see all security findings in one place (Security tab → Code Scanning) regardless of which tool found them.

### Defence in Depth (applied to CI/CD)
```
Layer 1: Local  → Gitleaks (pre-commit)
Layer 2: CI/CD  → TruffleHog + Semgrep + Trivy
Layer 3: Registry → Cosign signature verification
Layer 4: K8s    → OPA Gatekeeper admission control
Layer 5: Runtime → Falco runtime threat detection
```
